<task type="auto">
  <name>Task 1: Fix RLS SELECT policies — drop deleted_at IS NULL from USING clause on all 5 soft-delete tables</name>
  <files>supabase/migrations/20260223120000_fix_select_rls_soft_delete.sql</files>
  <action>
Create a new Supabase migration at `supabase/migrations/20260223120000_fix_select_rls_soft_delete.sql`.

Root cause: Supabase's implicit `RETURNING *` on UPDATE causes Postgres to re-evaluate the SELECT policy against the post-UPDATE row. For soft-delete, the row now has `deleted_at` set — so the SELECT policy's `deleted_at IS NULL` condition fails, producing "new row violates row-level security policy". The app already filters `deleted_at IS NULL` at the query level (via `.is('deleted_at', null)` in every Server Action); the SELECT policy does not need to enforce it.

For each of the 5 soft-delete tables — organizations, contacts, deals, interactions, tasks — drop the existing SELECT policy and recreate it WITHOUT `deleted_at IS NULL` in the USING clause. Only the `private.is_account_member(account_id)` check remains.

Current SELECT policies (from `20260222102951_create_rls_policies_for_crm_tables.sql`):
- `"organizations_select"`: `USING (deleted_at IS NULL AND private.is_account_member(account_id))`
- `"contacts_select"`: `USING (deleted_at IS NULL AND private.is_account_member(account_id))`
- `"deals_select"`: `USING (deleted_at IS NULL AND private.is_account_member(account_id))`
- `"interactions_select"`: `USING (deleted_at IS NULL AND private.is_account_member(account_id))`
- `"tasks_select"`: `USING (deleted_at IS NULL AND private.is_account_member(account_id))`

Write the migration as:

```sql
-- Migration: fix_select_rls_soft_delete
-- Remove deleted_at IS NULL from SELECT policies on soft-delete tables.
-- Supabase's implicit RETURNING * on UPDATE re-evaluates SELECT policy against
-- the new row state — soft-delete sets deleted_at, which then fails the IS NULL
-- check. App queries already filter deleted_at IS NULL themselves.

-- ORGANIZATIONS
DROP POLICY "organizations_select" ON public.organizations;
CREATE POLICY "organizations_select" ON public.organizations
  FOR SELECT TO authenticated
  USING (private.is_account_member(account_id));

-- CONTACTS
DROP POLICY "contacts_select" ON public.contacts;
CREATE POLICY "contacts_select" ON public.contacts
  FOR SELECT TO authenticated
  USING (private.is_account_member(account_id));

-- DEALS
DROP POLICY "deals_select" ON public.deals;
CREATE POLICY "deals_select" ON public.deals
  FOR SELECT TO authenticated
  USING (private.is_account_member(account_id));

-- INTERACTIONS
DROP POLICY "interactions_select" ON public.interactions;
CREATE POLICY "interactions_select" ON public.interactions
  FOR SELECT TO authenticated
  USING (private.is_account_member(account_id));

-- TASKS
DROP POLICY "tasks_select" ON public.tasks;
CREATE POLICY "tasks_select" ON public.tasks
  FOR SELECT TO authenticated
  USING (private.is_account_member(account_id));
```

<task type="auto">
  <name>Task 3: Fix deal form useEffect double-fire — remove onSuccess from dependency array</name>
  <files>src/components/deals/deal-form.tsx</files>
  <action>
Root cause: `useEffect` in `DealForm` has `[state, onSuccess]` as its dependency array (line 76). When `onSuccess(state.deal)` calls `setOpen(false)` in `DealCreateButton`, React re-renders `DealCreateButton` with a new `handleSuccess` function reference. This triggers the effect again while `state.success` is still truthy — `onSuccess` fires a second time, prepending the same deal twice into the Kanban array, causing dnd-kit's SortableContext to throw a duplicate-key crash.

**`src/components/deals/deal-form.tsx` — line 76:**
Change:
```ts
  }, [state, onSuccess])
```
To:
```ts
  }, [state])
```

That is the only change. One character removed. The effect runs only when `state` changes — which is the correct trigger (the Server Action returned a new result). `onSuccess` is captured via closure at the time `state` changes, which is always the current version.
